# Implementation Plan: Repository Storage Identity and Path Strategy

**Branch**: `010-repo-storage-identity` | **Date**: 2026-05-26 | **Spec**: [spec.md](spec.md)

## Summary

Introduce a two-layer repository identity model — an external `namespace/repo-name` path and an internal stable `repo_id` (UUIDv7) — so rename and transfer operations update only database mappings without moving storage. `gitstore-api` owns the `(namespace_id, name) → repo_id` lookup; `gitstore-git-service` receives only pre-resolved `repo_id` values and derives storage paths using a deterministic two-level hex-prefix fanout formula.

## Implementation Phases

### Phase 1 — Contracts (blocks everything)

**Constitution gate**: API-First — contracts must be committed before any implementation.

1. Update `shared/proto/gitstore/git/v1/git_service.proto` from contract file.
2. Merge `graphql.repository.graphqls` into `graph/schema.graphqls` (or `shared/schemas/`).
3. Update `001_initial_schema.cql` and `002_add_initial_indices.cql` in place (no new files; see research D-003).
4. Regenerate gRPC Go bindings (`protoc`).
5. Regenerate gqlgen model/resolver stubs.

---

### Phase 2 — Go data layer (US1, US2, US3 foundation)

**Test-first**: write failing tests before each implementation step.

1. Add `Repository` and `NamespaceMapping` structs to `entities.go`.
2. Add interface methods to `datastore.go`.
3. Implement memdb backend: schema tables + CRUD + `LookupRepository` + `RenameRepository` + `TransferRepository`.
4. Implement Scylla backend: models + CQL operations.

**Key invariants under test**:
- `LookupRepository(ns_id, name)` returns correct `repo_id`.
- After `RenameRepository`, old name returns `ErrNotFound`, new name returns original `repo_id`.
- After `TransferRepository`, old namespace returns `ErrNotFound`, new namespace returns original `repo_id`.
- Namespace slug rename requires zero `NamespaceMapping` row changes.

---

### Phase 3 — Rust fanout path resolver (US2 foundation)

**Test-first**: unit tests before implementation.

1. Extract `resolve_repo_path` into `git/repo.rs` as `fanout_path(data_dir, repo_id)`.
2. Validate UUID format (36-char hyphenated hex); return `Status::invalid_argument` on malformed input.
3. Apply two-level fanout formula.
4. Wire into all gRPC handlers (`CreateRepository`, `DeleteRepository`, all read/write/tag RPCs).

**Key invariants under test**:
- Same `repo_id` always produces same path (stability).
- Distinct UUIDs never produce the same path (collision-free).
- Malformed `repo_id` (contains `/`, `..`, empty, wrong length) returns `INVALID_ARGUMENT`.
- `CreateRepository` creates the fanout directory structure and returns the path in `storage_path`.

---

### Phase 4 — GraphQL resolvers (US1, US2, US3)

**Test-first**: contract tests against schema before resolver implementation.

1. Implement `createRepository` mutation: create `Repository` record + `NamespaceMapping` record + gRPC `CreateRepository` call.
2. Implement `repository` query: resolve `RepositoryBy.namespacePath` via `LookupRepository`.
3. Implement `repositories` query: list by namespace.
4. Implement `renameRepository` mutation: `RenameRepository` in datastore (no gRPC call needed — storage unchanged).
5. Implement `transferRepository` mutation: `TransferRepository` in datastore + update `Repository.NamespaceID` (no gRPC call needed).
6. Implement `deleteRepository` mutation: delete mapping + delete repository record + gRPC `DeleteRepository` call.

**Structured logging required** (Principle IV) on:
- Each `LookupRepository` call (namespace_id, name, resolved repo_id or not-found).
- Each `RenameRepository` / `TransferRepository` (old path, new path, repo_id).
- gRPC delegation calls (repo_id, RPC name, outcome).

---

### Phase 5 — Migration tooling (FR-011)

1. Write a `cmd/migrate-storage/main.go` one-shot tool that:
   - Reads existing bare repos from `data_dir` (current flat layout: `{data_dir}/{repo_name}.git`).
   - For each repo: assigns a new UUIDv7, creates `Repository` + `NamespaceMapping` records, moves the directory to the fanout path.
   - Logs every step and runs `git fsck` on each moved repo.
2. No zero-downtime requirement. Tool is run against a stopped service.

---

   - Common failure modes: orphaned storage, missing mapping, malformed UUID.
2. Update `quickstart.md` (this spec) with any deviations found during implementation.
