# Research: Decouple API from Git Storage via gRPC Git Service

**Feature**: 004-grpc-git-service
**Date**: 2026-05-06

## Decision 1 ‚Äî Proto Toolchain

**Decision**: buf CLI with `buf.yaml` (v2 workspace) at repo root; `.proto` files under `shared/proto/gitstore/git/v1/`. Per-language `buf.gen.yaml` files generate into `gitstore-git-service/gen/` (Rust) and `gitstore-api/gen/` (Go). Generated code is committed to source control.

**Rationale**: buf enforces breaking-change detection (`buf breaking`), handles lint, and generates both Rust and Go code reproducibly from a single `buf.gen.yaml`. Single source of truth at `proto/` eliminates cross-service drift. `buf breaking --against .git#branch=main` can be added as a CI step.

**Alternatives considered**: Raw `protoc` + Makefile ‚Äî no breaking-change detection, fragile across developer machines. Copying `.proto` into each service ‚Äî causes drift. Separate proto git submodule ‚Äî adds operational overhead for an internal contract.

---

## Decision 2 ‚Äî Rust gRPC Server

**Decision**: `tonic` 0.12 + `prost` 0.13 + `tonic-build` in `build.rs`.

**Rationale**: `tonic` is the de-facto async gRPC library for Rust/tokio (already the async runtime in git-service). `prost` is tonic's native codec with zero-copy-friendly generated structs. `tonic-build` in `build.rs` runs codegen at compile time and keeps generated code reproducible.

**Alternatives considered**: `buf generate` with Rust plugins ‚Äî delegates to `tonic-build` anyway, so `build.rs` is simpler. `grpcio` (C binding) ‚Äî heavier, not idiomatic for Tokio. `h2` direct ‚Äî no codegen, no streaming support.

---

## Decision 3 ‚Äî Go gRPC Client

**Decision**: `google.golang.org/grpc` v1.x + `google.golang.org/protobuf` + buf-generated Go stubs (via `protoc-gen-go` + `protoc-gen-go-grpc` in `buf.gen.yaml`).

**Rationale**: Official Google-maintained packages, idiomatic with Go modules. `gitstore-api` has no existing gRPC deps, so no migration cost. The older `github.com/golang/protobuf` is deprecated; `google.golang.org/protobuf` is its successor.

**Alternatives considered**: `connectrpc/connect-go` ‚Äî different wire format (not native gRPC), rejected. Manual `protoc` ‚Äî fragile and non-reproducible.

---

## Decision 4 ‚Äî Per-RPC Prometheus Metrics

**Decision**:
- Rust server: `tonic-prometheus-layer` crate; Tower layer wrapping the tonic service, recording `grpc_server_handled_total` (counter, labels: method, status) and `grpc_server_handling_seconds` (histogram per method)
- Go client: `github.com/grpc-ecosystem/go-grpc-middleware/v2` with `grpcprom` package; `grpcprom.UnaryClientInterceptor` + `grpcprom.StreamClientInterceptor` wired into `grpc.NewClient`; exposes `grpc_client_*` metrics family on the existing Prometheus endpoint

**Rationale**: Interceptor/middleware pattern keeps metrics instrumentation out of business logic. Consistent label set allows correlation between server and client dashboards.

**Alternatives considered**: Manual `prometheus.CounterVec` in each handler ‚Äî more flexible but significant boilerplate. `otel` with Prometheus bridge ‚Äî adds OTEL dependency not yet present in the project.

---

## Decision 5 ‚Äî Integration Testing

**Decision**: Two-tier approach:
1. **Unit/contract (Go-side only)**: In-process Go gRPC stub server using `google.golang.org/grpc/test/bufconn` ‚Äî fast, no Docker, validates Go client behaviour and error handling
2. **Cross-language integration**: `testcontainers-go` (v0.35+) starts the real `gitstore-git-service` Docker image in `TestMain`; Go API client dials it; tests validate end-to-end RPC round-trips against a real git repository

**Rationale**: `bufconn` is fast for catching Go-side regressions. `testcontainers-go` with a real compiled server is required for FR-008 (contract tests covering both read and write paths) ‚Äî a Go stub cannot catch Rust serialisation or git operation failures.

**Alternatives considered**: `docker compose` in CI ‚Äî harder to orchestrate deterministically from `go test`. Mocking the gRPC interface ‚Äî rejected per constitution Principle I (test-first against real services).

---

## Decision 6 ‚Äî Contract Versioning

**Decision**: Proto package namespace `gitstore.git.v1`; Go option `go_package = "github.com/gitstore-dev/gitstore/api/gen/gitstore/git/v1;gitv1"`; Rust module `gitstore_git_v1` (generated by tonic-build). Version field `string schema_version = 1` in `ServiceInfo` message returned by a `GetServiceInfo` RPC ‚Äî server validates caller's declared version.

**Rationale**: `package gitstore.git.v1` is the [Buf/Google API style guide](https://buf.build/docs/best-practices/style-guide) recommendation. The `v1` in the package namespace makes `buf breaking` enforcement automatic; a breaking change requires a new `v2` package, which enforces the atomic deployment assumption documented in the spec.

**Alternatives considered**: Version only in service name (`GitServiceV1`) ‚Äî makes `buf breaking` harder to enforce and clutters generated type names. No versioning ‚Äî rejected per spec clarification Q2.
